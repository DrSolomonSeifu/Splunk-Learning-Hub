Section 3.1

Explain Fields

What Are Fields?

Fields are searchable name/value pairs in event data — they are knowledge objects that let you pinpoint specific pieces of information inside events. For example: host=www2, status=200, user=admin.

Searching with fields is more precise and more efficient than searching with keywords alone. Instead of hoping Splunk finds "200" somewhere in the raw text, you can say "give me events where the status field equals 200."

Two Types of Fields

TypeWhen ExtractedExamples
Indexed Fields (Index-Time)When data is ingested — stored in the indexhost, source, sourcetype, index, linecount, punct, splunk_server, timestamp
Search-Time FieldsWhen you run a search — extracted on the flystatus, clientip, action, product_name, user
Key Concept — Index-Time Fields

When data is indexed, Splunk automatically creates default fields for every event:

  • host — hostname or IP where the event originated
  • source — the input source or file path
  • sourcetype — the format/type of data
  • index — which index stores the event

These fields exist before you even search — they're part of the stored data.

Key Concept — Search-Time Fields

At search time, Splunk looks for key=value patterns in the raw event data and automatically extracts them into field-value pairs. For example:

  • If the raw data contains action=purchase, Splunk creates a field called action with value purchase
  • Sourcetype definitions help Splunk recognize patterns even without explicit key=value format — for example, access_combined interprets 203.45.206.135 as clientip=203.45.206.135

How Search Modes Affect Field Extraction

The search mode determines which fields are extracted at search time:

ModeField DiscoveryBehavior
FastDisabledReturns only default fields + fields explicitly in your search. With transforming commands, shows Statistics/Visualization only.
VerboseFullExtracts ALL fields from every event. Always shows Events tab + timeline. Slowest mode.
Smart (default)AdaptiveNon-transforming → behaves like Verbose. Transforming → behaves like Fast. Best of both worlds.

    SPL

  3. Now run a transforming search in all three modes:

  4. In Fast and Smart mode — can you see the Events tab?
  5. In Verbose mode — can you still see the Events tab?
📝 Field counts per mode

📋 Section 3.1 Key Takeaways

  • Fields = searchable name/value pairs (knowledge objects)
  • Indexed fields (host, source, sourcetype, index) are created at index time
  • Search-time fields are extracted when you run a search
  • Search mode controls which fields are extracted: Fast (minimal), Verbose (all), Smart (adaptive)

Section 3.2

Use the Fields Sidebar

What Is the Fields Sidebar?

The Fields Sidebar is the panel on the left side of the search results. It lists all fields available in your current search results and lets you explore, filter, and interact with your data without writing additional SPL.

Three Sections of the Fields Sidebar

SectionWhat It ShowsDetails
Selected FieldsFields displayed for every eventDefault: host, source, sourcetype. You can add more.
Interesting FieldsFields in at least 20% of eventsAuto-detected. Shows field name + unique value count.
All Fields (link)Every field including rare onesClick to open a window showing all fields, types, and statistics.

Field Type Symbols

SymbolMeaningExample
𝛼 (alpha)Alphanumeric (text) valueshost, sourcetype, action
# (hash)Numeric valuesstatus, bytes, price

The number next to each field shows how many unique values exist for that field in the current results.

The Field Window

Clicking any field name in the sidebar opens the Field Window, which lets you:

  • See the top values for that field
  • Click a value to add it as a filter to your search
  • Create quick reports — Splunk adds transforming commands automatically

Report options differ by field type: numeric fields offer average, min, max, etc. Alphanumeric fields offer top values and count by.

⚠️ Best Practice — Always Specify an Index

If no index is specified in your search, Splunk returns data from all indexes accessible to your role. Always include an index to keep searches fast and focused.

📋 Section 3.2 Key Takeaways

  • Fields Sidebar: Selected Fields, Interesting Fields (20%+ of events), All Fields
  • 𝛼 = alphanumeric, # = numeric
  • Click a field to open the Field Window — explore values & create quick reports
  • Reports auto-generate different options for text vs. numeric fields
  • Always specify an index in your searches

Section 3.3

Use Fields in Searches

Fields as Filters

Including fields in your search lets you filter events precisely. Without a field name, Splunk searches the entire raw event — which can produce unexpected matches.

Example problem: Searching for just 503 could match both status=503 (HTTP error) AND area_code=503 (Portland phone number). Use status=503 to get only what you want.

⚠️ Case Sensitivity Reminder

Field names ARE case-sensitive: status ≠ Status

Field values are NOT case-sensitive: status=OK = status=ok

Quotation Marks for Phrases

When searching for values that contain spaces or special characters, wrap them in double quotation marks:

SPL
product_name="Dream Crusher"    ← matches exact phrase
product_name=Dream Crusher      ← WRONG: searches product_name=Dream AND keyword "Crusher"
🧪 Lab 4

Fields as Filters

  1. Search without specifying a field:
    SPL
    index=_internal ERROR
  2. Now search with a field:
    SPL
    index=_internal log_level=ERROR
  3. Compare the event counts — which is more precise?
  4. Test case sensitivity:
    SPL
    index=_internal log_level=ERROR
index=_internal log_level=error

    Do both return the same results? (They should — values are case-insensitive)

Wildcard Best Practices

The asterisk (*) matches any number of characters, but how you use it matters:

PracticeExampleWhy
✅ End of term (best)fail*Efficient — Splunk scans forward
⚠️ Beginning of term (avoid)*failSlow — must check every string
❌ Middle of term (avoid)f*ilInconsistent with punctuation indexing
❌ Matching punctuation/cart*Unreliable — specify exact paths instead

Rule of thumb: A specific search is always more efficient than one with wildcards.

📝 CySA+ Practice

According to Splunk best practices, which wildcard placement is most efficient?

  • A. f*il
  • B. *fail
  • C. fail*
  • D. *fail*
✅ Correct: C — Wildcards at the end of a term are the most efficient. Prefix wildcards (*fail) must scan every string, dramatically slowing the search.
🧪 Lab 5

index=_internal sourcetype=*splunk*
index=_internal sourcetype=*splunkd

Comparison Operators

Beyond =, Splunk supports comparison operators to match ranges of values:

OperatorMeaningExample
=Equal tostatus=200
!=Not equal tostatus!=200
>Greater thanbytes>1000
<Less thanbytes<500
>=Greater than or equalstatus>=400
<=Less than or equalstatus<=299
🧪 Lab 6

Comparison Operators in Action

  1. Find events where a numeric field exceeds a threshold:
    SPL
    index=_internal linecount>5
  2. Find events within a range:
    SPL
    index=_internal linecount>=3 linecount<=10

The Difference Between != and NOT

Both exclude events, but they work differently:

MethodWhat It ReturnsImportant Difference
status!=404Events where status has a value that is NOT 404Excludes events that don't have a status field at all
NOT status=404All events EXCEPT those with status=404Includes events that don't have a status field
⚠️ Performance Tip

Using != or NOT to exclude events is less efficient than including what you want. Inclusion is always better than exclusion. When possible, search for the values you want rather than excluding the ones you don't.

    SPL
    index=_internal log_level!=ERROR
  2. Search with NOT:
    SPL
    index=_internal NOT log_level=ERROR
  3. Compare event counts — which returned more? Why?

📝 CySA+ Practice

What operator is implied between search terms unless otherwise specified?

  • A. AND
  • B. OR
  • C. NOT
  • D. LIKE
✅ Correct: A — AND is the default operator. error webserver is the same as error AND webserver.

📋 Section 3.3 Key Takeaways

  • Always use field names for precise filtering — avoid bare keywords
  • Wildcards at the end of a term = most efficient
  • Comparison operators: =, !=, >, <, >=, <=
  • != excludes events without the field entirely; NOT includes them
  • Inclusion is more efficient than exclusion

Section 3.4

Preview: The Table and Rename Commands

The Pipe Operator ( | )

In Splunk, the vertical bar | is the pipe operator. It chains commands together in a search pipeline — the output of one command becomes the input for the next.

Format
search command | transforming command | another command

The table Command

The table command displays specific fields in a clean tabular format. List the fields you want to see, separated by spaces:

SPL
index=_internal sourcetype=splunkd | table _time host log_level component message

This is a transforming command — results appear on the Statistics tab.

The rename Command

The rename command gives fields more readable names. Use the as keyword:

SPL
... | table clientip product_name price | rename clientip as "Client IP" product_name as "Product" price as "Price ($)"

Use double quotes for names with spaces. The rename only lasts for the current search — it does not change the indexed data.

⚠️ Important

Once a field is renamed, you must use the new name for the rest of the search pipeline. The original field name no longer works after the rename command.

🧪 Lab 8

Build a Table

  1. Create a basic table:
    SPL
    index=_internal sourcetype=splunkd log_level=ERROR | table _time host component message
  2. Observe: results appear on the Statistics tab
  3. Try adding/removing fields from the table command — how does the output change?
🧪 Lab 9

Rename Fields

  1. Run a table with renamed fields:
    SPL
    index=_internal sourcetype=splunkd log_level=ERROR
| table _time host component message
| rename _time as "Timestamp" host as "Server" component as "Module" message as "Details"
  2. Notice: column headers now show the friendly names
  3. Try adding a sort command after rename — use the new name:
    SPL
    ... | sort -"Timestamp"
🧪 Lab 10

Real-World Scenario: Incident Summary Table

Scenario: Your manager asks for a table showing recent errors with the server name, component, and description.

  1. Build the search:
    SPL
    index=_internal log_level=ERROR earliest=-1h
| table _time host component message
| rename _time as "Time" host as "Server" component as "Component" message as "Error Description"
  2. Click the paintbrush icon on the Statistics tab to format the table (column colors, number formatting)
  3. Export the table as CSV

📋 Section 3.4 Key Takeaways

  • The pipe | chains commands in a search pipeline
  • table displays specific fields in tabular format (Statistics tab)
  • rename gives fields friendly names using as
  • Renamed fields must be referenced by their new name in subsequent commands
  • Renaming is temporary — it doesn't change indexed data

Lesson Review — Test Yourself

1. What are fields in Splunk? Provide an example.▼

Fields are searchable name/value pairs in event data. Example: status=200, host=webserver01

2. What is the difference between indexed fields and search-time fields?▼

Indexed fields are extracted when data is ingested (host, source, sourcetype). Search-time fields are extracted when you run a search from key=value patterns or sourcetype definitions.

3. What are the three search modes and how do they differ?▼

Fast = speed priority, minimal fields. Verbose = completeness priority, all fields. Smart (default) = adaptive — Verbose for basic searches, Fast for transforming searches.

4. What does the 𝛼 symbol next to a field name indicate?▼

The field contains alphanumeric (text) values.

5. What does the # symbol next to a field name indicate?▼

The field contains numeric values.

6. What are the three default Selected Fields?▼

host, source, sourcetype

7. What character does Splunk use as a wildcard?▼

The asterisk (*). Best placed at the end of a term for efficiency.

8. What operator is implied between search terms?▼

AND. error webserver = error AND webserver.

9. After renaming a field, how long does the change exist?▼

Only for the lifetime of the current search. No permanent change to indexed data.
