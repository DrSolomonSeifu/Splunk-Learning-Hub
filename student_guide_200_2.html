Section 2.0

Adding Data to Splunk

How Does Data Get Into Splunk?

Before you can search, you need data inside Splunk. Splunk supports logs, metrics, and events — both structured (CSV, JSON) and unstructured (raw log files).

Data is typically added by IT or DevOps teams who configure forwarders, file monitors, and network inputs. End-users usually don't add data, though exceptions exist in lab settings.

Three Methods to Add Data

MethodDescriptionBest For
UploadUpload a file directly (.log, .csv)Lab practice, sample datasets
MonitorWatch a file or folder path continuouslyLive logs that update frequently
ForwardUse a Universal Forwarder agent on remote machinesProduction, multi-system environments
Key Concept — Indexing

An index is where Splunk stores your data after processing. During indexing, Splunk breaks data into individual events, extracts fields, and assigns timestamps. Default indexes include: main, _internal, and _audit.

Key Concept — Sourcetype

A sourcetype is a metadata field that tells Splunk the format of your data. It controls how Splunk parses events and extracts fields. Examples: access_combined (Apache logs), syslog (system logs), json (structured data).

📋 Key Takeaways

  • Data must be directed to an index and tagged with a sourcetype
  • The sourcetype helps Splunk parse and extract fields correctly
  • Methods: Upload, Monitor, or Forward
  • Think of indexing as the storage box, and sourcetype as the label that tells Splunk how to read it

Run Basic Searches

Searching with SPL

Searching means using the Search & Reporting app to query indexed data with SPL (Search Processing Language). You can search for keywords, phrases, or field-value pairs and filter by time, index, sourcetype, and more.

The Search Interface

6Table ViewsPoint-and-click search builder
🧪 Lab 1

Your First Search

  1. Open your Splunk instance → click Search & Reporting
  2. In the search bar, type and press Enter:
    SPL
    index=_internal
  3. Set the time range to "Last 15 minutes"
  4. Now search with a keyword:
    SPL
    index=_internal error
  5. Observe the results and identify the fields shown in each event

Key-Value Pairs

Key-value pairs link a field name (key) to its data (value) using the equal sign (=).

Format
key=value
source=/var/log/syslog
status=404
user=admin
⚠️ Remember

Keys ARE case-sensitive: source ≠ Source — these return different results!

Values are NOT case-sensitive: source=/var/log/syslog = source=/VAR/LOG/SYSLOG

🧪 Lab 2

Case Sensitivity Test

  1. Run with lowercase:
    SPL
    index=_internal source=*splunkd.log
  2. Run with uppercase:
    SPL
    index=_internal Source=*splunkd.log
  3. Compare: Did the uppercase version return fewer or no events?
📝 Your Observations
📝 CySA+ Practice

What syntax is used to link key/value pairs in search strings?

  • A. action+purchase
  • B. action=purchase
  • C. action | purchase
  • D. action equal purchase
✅ Correct: B — Key-value pairs use the equal sign (=) to link the field name with its value.

Wildcards

The asterisk (*) matches zero or more characters. Use it when you don't know the exact full value.

SPL
product_name=web*        → matches web, website, webinar
user=*admin              → matches superadmin, testadmin
source=*syslog*          → matches anything containing "syslog"

⚠️ Wildcards at the beginning of a value can slow searches. Use at the end when possible.

🧪 Lab 3

Wildcard Searches

  1. Try these searches and compare results:
    SPL
    index=_internal source=*log*
index=_internal sourcetype=splunkd*
index=_internal component=*
  2. Remove or modify the wildcards — how do results change?

Search Operators: AND, OR, NOT

OperatorWhat It DoesExample
ANDBoth terms must match (default if omitted)error AND webserver
OREither term can matcherror OR warning
NOTExclude matching eventserror NOT database

Default behavior: error webserver = error AND webserver (AND is implied).

🧪 Lab 4

Boolean Operators

  1. Test AND (implicit vs. explicit):
    SPL
    index=_internal error splunkd
index=_internal error AND splunkd

    Do both produce the same results?

  2. Test OR:
    SPL
    index=_internal error OR warn
  3. Test NOT:
    SPL
    index=_internal error NOT metrics
📝 Compare: Which returned more results — AND or OR?
📝 CySA+ Practice

Which search matches events containing BOTH "error" AND "webserver"?

  • A. index=security Error Webserver
  • B. index=security error OR webserver
  • C. index=security "error web server"
  • D. index=security NOT error NOT webserver
✅ Correct: A — Splunk assumes AND between terms. B returns either. C is an exact phrase. D excludes both.

Helpful Search Tools

  • Search Assistant — suggests terms, field names, and commands as you type
  • Search History — view and rerun previous searches (found under the search bar)
  • Data Summary — browse available hosts, sourcetypes, and data sources (How to Search → Data Summary)

📋 Section 2.1 Key Takeaways

  • Key-value pairs use = — keys are case-sensitive, values are not
  • AND is the default operator between terms
  • * wildcard matches any number of characters
  • Use quotation marks for values with spaces: status="server error"

Section 2.4

Refine Searches

How to Refine

  • Add more search terms to be specific
  • Use field-value filters: status=404, src_ip=10.0.0.1
  • Drill down — expand events using the ⓘ icon
  • Click field names for statistics and unique values
  • Click field values to add/remove them from your search
  • Change search mode (Fast, Smart, Verbose)

Three Search Modes

ModeSpeedDetailBest For
Fast⚡ FastestMinimal fieldsQuick counts, dashboards
Smart (default)⚡⚡ BalancedAuto-adjustsGeneral searching
Verbose🐢 SlowestAll fieldsTroubleshooting, forensics
🧪 Lab 7

Progressive Refinement

  1. Start broad (note event count):
    SPL
    index=_internal
  2. Add sourcetype:
    SPL
    index=_internal sourcetype=splunkd
  3. Add keyword:
    SPL
    index=_internal sourcetype=splunkd log_level=ERROR
  4. Watch the count drop at each step
  5. Drill-down: Expand an event with ⓘ → click the timestamp → "Show events before/after this time"
  6. Search modes: Run the same search in Fast, Smart, and Verbose — compare speed and fields extracted
📝 Event count at each step

Section 2.5

Use the Timeline

Timeline as an Investigation Tool

The timeline is a bar chart showing event frequency over time. Use it to spot peaks (spikes/attacks) and valleys (downtime). Hover over bars for counts; click to drill down.

⚠️ Important Distinction

Selecting a range on the timeline → filters existing results (no new search)

Zoom In/Out or Zoom to Selection → runs a NEW search with updated time frame

Timeline Format Options

OptionDescription
Hidden / Compact / FullControl timeline display size
Linear ScaleProportional bar heights
Log ScaleReveals small spikes next to large ones
Zoom to SelectionClick-drag across bars to zoom in
🧪 Lab 8

Investigate with the Timeline

  1. Run:
    SPL
    index=_internal log_level=ERROR
  2. Hover over bars to see event counts
  3. Click and drag to select bars → click Zoom to Selection
  4. Toggle between Compact and Full view
  5. Switch between Linear and Log scale
  6. Click Deselect to return to full view

Use Time Modifiers in Your Search

Absolute vs. Relative Time

TypeFormatExample
Absolute%m/%d/%Y:%H:%M:%Searliest=04/19/2023:00:00:00 latest=04/27/2023:00:00:00
Relative-[number][unit]earliest=-60m latest=now

Relative Time Units

UnitMeaningUnitMeaning
sSecondsdDays
mMinuteswWeeks
hHoursmonMonths
⚠️ Override Rule

Time modifiers in the search bar always override the Time Range Picker. If the picker says "Last 24 hours" but you type earliest=-30m, only 30 minutes are searched.

Snap to Time with @

Use @ to round time to the nearest boundary:

  • earliest=-1h@h — 1 hour ago, snapped to top of hour (15:45 → 15:00)
  • earliest=-7d@d — 7 days ago, snapped to midnight
  • earliest=-mon@mon latest=@mon — entire previous month
🧪 Lab 9

Time Modifiers in Practice

  1. Relative time:
    SPL
    index=_internal earliest=-15m
  2. Override test — set Picker to "Last 24 hours", then run:
    SPL
    index=_internal earliest=-10m latest=now

    How much time was actually searched?

  3. Snap to time:
    SPL
    index=_internal earliest=-1h@h latest=now
📝 What start time did the snap-to-time produce?

Section 2.7

Control a Search Job

What Is a Search Job?

Every search creates a search job — a process tracking the owner, app, event count, and duration. Default lifetime: 10 minutes (extendable to 7 days).

Job Management Tools

ToolPurposeAccess
Job InspectorExecution costs & diagnosticsJob → Inspect Job
Job DetailsPerformance overviewVia Job Inspector
Jobs ManagerView all recent jobsActivity → Jobs
Job MenuEdit, background, inspect, deleteClick "Job" on search page
🧪 Lab 10

Explore Search Jobs

  1. Run:
    SPL
    index=_internal | stats count by sourcetype
  2. Click Job → Inspect Job — review execution costs
  3. Go to Activity → Jobs — find your search job
  4. Click Job → Edit Job Settings — try extending the lifetime

Knowledge Check

Lesson Review — Test Yourself

Click each question to reveal the answer.

1. What syntax links key/value pairs?▼

The equal sign (=). Example: status=404

2. What does "earliest=-72h@h latest=@d" do?▼

Starts 72 hours ago, snapped to the hour, and ends at the start of the current day.

3. What is the function of the timeline?▼

A visual representation of event distribution over time, helping identify patterns and anomalies.

4. What can be configured using Edit Job Settings?▼

Read permissions, job lifetime, and a shareable URL.

5. How long does Splunk retain a search job by default?▼

10 minutes. Extendable to 7 days.

6. Which Boolean operator is implied between terms?▼

AND. error webserver = error AND webserver.

7. What character is the wildcard?▼

The asterisk (*).

8. What is the easiest way to optimize searches?▼

Be specific with search terms and narrow the time range.

9. What is an event?▼

A single timestamped data entry representing a record of activity.

10. What are the three search modes?▼

Fast (speed), Smart (balanced, default), Verbose (full detail).
