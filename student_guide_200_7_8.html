Section 7.1

What Is a Lookup?

The Problem — Data Without Context

Machine data contains valuable insights, but it doesn't always provide the full picture. Your event data might show user_id=101, but that alone doesn't tell you who that user is, what department they belong to, or what role they have.

The Solution — Lookups

A lookup in Splunk enriches or transforms your data by bringing in information from external sources, such as tables or files.

  • Enrich results with extra details — department names, locations, device owners
  • Match values in indexed data with corresponding values in an external table
  • Work through paired fields — at least one field must be common between event data and the lookup table
Quick Example

Your event has user_id=101 — that doesn't mean much.
A lookup file tells us 101 belongs to Alex from HR.

Real-World Use Case

A field productId=WC-98-047 means nothing alone. With a lookup file containing product_name, price, and category, Splunk can automatically pull that in.

SPL
index=main sourcetype="access_combined_wcookie"
| where isnotnull(productId)
| lookup product_price.csv productId OUTPUT product_name, price, sale_price
| table productId, product_name, price, sale_price

Section 7.2

Four Types of Lookups

TypeSourceBest ForExample
CSV Lookup.csv fileSmall, static datasetsProduct list, employee directory
External LookupPython scriptDynamic, real-time dataDNS lookup to resolve IPs
KV Store LookupSplunk NoSQL DBLarge, frequently changingDevice inventory updated daily
Geospatial LookupKML/KMZ filesLocation enrichmentMapping IPs to countries

Detailed Breakdown

  • CSV Lookup — Most common. Also called a static lookup. Upload a .csv file with field-value pairs.
  • External Lookup — Also known as a scripted lookup. Uses Python scripts to dynamically fetch data (e.g., DNS server).
  • KV Store Lookup — Splunk's internal Key-Value Store. Best for large or frequently changing data.
  • Geospatial Lookup — Uses KML (Keyhole Markup Language) or KMZ (compressed KML) files for location mapping.
Summary

CSV = simple & static | External = dynamic scripting | KV Store = large/changing | Geospatial = mapping & location

Section 7.3

Lookup Table Files

What Are Lookup Table Files?

The actual files storing lookup data — think of them as external reference sheets Splunk reads during searches.

  • Only CSV and Geospatial lookups require uploading a file
  • One file can support multiple lookup definitions
  • Example: Same product_price.csv → one lookup for names, another for prices

CSV File Restrictions

  • At least two columns — one lookup key + one data field
  • One column must match a field in your events
  • Plain ASCII or valid UTF-8 only
  • No old-style line endings (\r)
  • Header row under 4096 characters

View Existing Lookup Files

  3. View a file's contents:

Section 7.4

Lookup Definitions

What Is a Lookup Definition?

Tells Splunk where to find a lookup table and how to use it. Acts as a bridge between search commands and the actual file.

  • Assigns a name and path to the file
  • Can include matching rules or restrictions
  • A single table can have multiple definitions (one matches employee_id, another matches email)
  • All lookup types require a definition before use
Think of It This Way

Lookup Table File = the actual data (the CSV file)
Lookup Definition = the instructions for how Splunk uses that data

Section 7.5

Automatic Lookups

What Are Automatic Lookups?

Apply a lookup automatically — no need to type the lookup command every time.

  • Links a definition to specific fields in indexed data
  • Whenever those fields appear, Splunk enriches results automatically
  • Ensures consistency across all users and reports
Before vs After

Without: Must type | lookup http_codes status OUTPUT status_description status_type every time
With: Just run index=web and the enrichment fields appear automatically!

How to Configure

  1. Settings → Lookups → Automatic lookups → + Add new
  2. Choose destination app (e.g., search)
  3. Name it (e.g., http_codes)
  4. Select lookup definition
  5. Apply to sourcetype (e.g., access_combined)
  6. Map input fields (event field → lookup field)
  7. Map output fields (fields to add)
  8. Click Save
🧪 Lab 2

Configure an Automatic Lookup

  1. Go to Settings → Lookups → Automatic lookups
  2. Click + Add new
  3. App: search, Name: http_codes
  4. Lookup table: http_codes, Sourcetype: access_combined
  5. Input: status → status | Output: status_description, status_type
  6. Save and set Read permissions for Everyone
  7. Verify — run without lookup command:
    SPL
    index=main sourcetype="access_combined_wcookie"
| stats count by status_type, status_description

Section 7.7

CSV Lookups — Deep Dive

What Is a CSV File?

CSV = Comma-Separated Values. A plain text file that organizes data in tabular format.

  • Each line = a row of data
  • Values separated by commas
  • Easy for humans and machines to read
  • Standard format for Excel, databases, and Splunk

CSV Lookup Characteristics

  • File-based and static — the file rarely changes
  • Great for small datasets that don't change frequently
  • Examples: product lists, employee directories, HTTP status codes
🛠️
Section 7.8

Creating a CSV Lookup — Step by Step

Step 1: Upload the File

  1. Settings → Lookups → Lookup table files → + Add new
  2. Choose app (e.g., search)
  3. Click Choose File → select your CSV (e.g., http_status.csv)
  4. Set Destination filename (e.g., http_codes.csv)
  5. Click Save

Note: Your role must include upload_lookup_files capability.

Step 2: Set Permissions

  1. Find your file → click Permissions
  2. Choose This app only (search)
  3. Check Read for Everyone
  4. Click Save

Step 3: Create a Definition

  1. Settings → Lookups → Lookup definitions → + Add new
  2. Name it (e.g., http_codes)
  3. Select the lookup table file
  4. Click Save
🧪 Lab 4

Full Lookup Creation Workflow

  1. Download or create a CSV with columns: status, status_description, status_type
  2. Upload: Settings → Lookups → Lookup table files → + Add new
  3. Set Read permissions for Everyone
  4. Create a lookup definition pointing to it
  5. Verify:
    SPL
    | inputlookup http_codes.csv
  6. Test with a filter:
    SPL
    | inputlookup http_codes.csv | where status >= 400

Section 7.9

Using Lookups in Searches

Without a Lookup — Raw Data

SPL — No lookup
index=web
| stats count(status) as "Number of Requests" by status

Shows status codes (200, 404, 503) with no explanation of what they mean.

With a Lookup — Enriched Data

SPL — With lookup
index=main sourcetype="access_combined_wcookie"
| lookup http_status.csv status OUTPUT status_description status_type
| stats count(status) as "Number of Requests" by status status_description status_type
| rename status_type as "HTTP message category", status_description as "Message details"
| sort - "HTTP message category"

Now each code shows readable categories like "Client Error" and descriptions like "Not Found".

Verify an Automatic Lookup

SPL — Auto lookup verification
index=main sourcetype="access_combined_wcookie"
| stats count(status) as "Number of Requests" by status_type, status_description
| rename status_type as "HTTP message category", status_description as "Message details"
| sort - "HTTP message category"

If automatic lookup is configured, enrichment fields appear without using the lookup command!

Section 8.1

Introduction to Scheduled Reports & Alerts

Why Automate?

Instead of running the same searches manually, you can schedule reports to run at set intervals and create alerts that notify you when conditions are met.

  • Important insights delivered on time
  • Critical issues caught and acted upon quickly
  • Makes Splunk a proactive tool, not just a reactive one

What Is an Alert?

Alerts monitor data in real-time or on a schedule to help you respond quickly to important events.

  • Built on saved searches — when results match conditions, Splunk triggers the alert
  • Actions: send email, run script, post to webhook, create ticket
  • Example: If failed logins exceed 10 in 5 minutes → alert the security team
⚡
Section 8.2

Alert Workflow — Four Parts

The Four Components

  1. Search — What do you want to track? (failed logins, error codes, etc.)
  2. Alert Type — How often should Splunk check? (Scheduled or Real-time)
  3. Trigger Conditions & Throttling — When should it fire? Throttle to prevent alert fatigue.
  4. Alert Actions — What happens? (email, script, ticket, triggered alerts list)
SPL — Example alert search
search error
| stats count by host
| where count > 100

Section 8.4

Creating a Scheduled Alert

Scenario: Daily Sales Alert

Alert if daily online sales drop below 800.

SPL
index=web sourcetype=access_combined action=purchase
| stats count as "Online Sales"

Configuration Steps

  1. Run the search → click Save As → Alert
  2. Title: Sales_Alert_Less_than_800_items
  3. Permissions: Shared in App
  4. Alert type: Scheduled
  5. Schedule: Run every day at 23:00
  6. Expires: 24 hours
  7. Trigger: Number of Results less than 800
  8. Trigger Once: Yes
  9. Action: Add to Triggered Alerts, Severity: Medium
  10. Click Save

Section 8.5

Real-Time Alerts — Per-Result

What Are Real-Time Per-Result Alerts?

Continuously search for events and trigger every time a matching event occurs.

  • Critical for immediate monitoring — unauthorized access, system failures
  • Continuously scans the incoming data stream
  • Use throttling to prevent alert flooding

Example: HTTP Server Error Alert

SPL
index=web sourcetype=access_combined (host=www1 OR host=www2 OR host=www3) status>=500
  • Type: Real-time, Per result
  • Throttle: 1 hour (suppress per status code)
  • Action: Add to Triggered Alerts, Severity: High

Section 8.6

Real-Time Alerts — Rolling Time Window

What Is a Rolling Time Window?

Monitors events over a continuous time frame (e.g., last 5 minutes) and triggers when a condition is met within that window.

  • Detects patterns or trends rather than individual events
  • Example: Alert if >20 failed logins within 5 minutes

Example: Failed Login Detection

SPL
index=security sourcetype=linux_secure ("Failed password")
  • Title: SEC_Alert_20_failed_login
  • Type: Real-time
  • Trigger: Number of Results > 20 in 5 minutes
  • Throttle: Suppress for 30 minutes
  • Action: Add to Triggered Alerts, Severity: High
Per-Result vs Rolling Window

Per-Result: "Alert me every time someone fails to log in" → individual event
Rolling Window: "Alert me if 20+ failures in 5 minutes" → pattern detection

Section 8.8

Viewing & Managing Triggered Alerts

How to View Triggered Alerts

  1. Go to Activity → Triggered Alerts
  2. View all alerts triggered in past 24 hours (default)
  3. Filter by: All, Yours, or This App's

Triggered Alert Details

FieldDescription
TimeWhen the alert triggered
Fired alertsAlert name(s)
AppContext/app it runs from
TypeScheduled or Real-time
SeverityHigh, Medium, Low
ModePer-result or Digest

When Do Alerts Appear?

  • "Add to Triggered Alerts" action must be enabled
  • Alert must have triggered recently
  • Retention period still active (default: 24h, extendable to 7 days)
  • Record must not have been deleted

Useful SPL for Alert Investigation

SPL — Failed scheduled alerts
index=_internal sourcetype=scheduler status=failure
| stats count by savedsearch_name, user, app
SPL — Successful alerts
index=_internal sourcetype=scheduler action_name="alert_action" status=success
| stats count by savedsearch_name
SPL — List via REST API
| rest /services/alerts/fired_alerts
| table title, triggered_alerts, app, alert_type, severity, alert_mode

Knowledge Check

Lesson 200.8 Review

1. What are the two alert types?▼
Scheduled and Real-time alerts.
2. Scheduled alert vs scheduled report?▼
A scheduled alert acts only when conditions are met. A scheduled report acts every time it completes.
3. Advantage and disadvantage of real-time alerts?▼
Advantage: Immediate detection. Disadvantage: Higher resource usage and potential alert fatigue.
4. What scheduling options exist for scheduled alerts?▼
Predefined (hourly, daily at specific time) or custom cron expressions.
5. Per-result vs rolling window trigger?▼
Per-result: triggers on each matching event. Rolling window: triggers based on conditions within a defined time window.
6. When should an admin throttle an alert?▼
To avoid repeated alerts for the same issue — e.g., suppress for 1 hour.
7. Conditions for alert to appear on Triggered Alerts page?▼
"Add to Triggered Alerts" enabled, triggered recently, retention active, not deleted.
8. What does severity level provide?▼
Tags alert with High/Medium/Low to help filter and prioritize on the Triggered Alerts page.

📋 Lesson 200.8 Summary

  • Alerts automate monitoring — built on saved searches with trigger conditions
  • Two types: Scheduled and Real-time (Per-result + Rolling window)
  • Throttling prevents alert fatigue by suppressing duplicates
  • Reports always run; Alerts only trigger when conditions are met
  • Manage alerts via Activity → Triggered Alerts
