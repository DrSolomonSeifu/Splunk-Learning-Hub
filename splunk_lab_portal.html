Lesson 200.1 — Splunk Basics

Get familiar with the Splunk interface, understand its architecture, and explore your first data.

Navigate Splunk WebUnderstand ComponentsExplore Data Summary
1.1

Navigate the Splunk Interface

Explore the home screen, apps, and main navigation areas

  1. Log in to Splunk Web (typically http://localhost:8000)
  2. Identify the Apps Bar at the top — click Search & Reporting
  3. Locate these areas: Search Bar, Time Range Picker, Data Summary button
  4. Click Data Summary and explore the three tabs: Hosts, Sources, Sourcetypes
  5. Write down 3 hosts and 3 sourcetypes you see listed
💡 Tip

The Data Summary button gives you a quick snapshot of ALL data in your Splunk instance — use it anytime you want to know what's available to search.

1.2

Explore Splunk Apps & Settings

Discover installed apps and basic configuration areas

  1. From the home screen, click the gear icon next to Apps to view all installed apps
  2. Go to Settings → Data Inputs — explore the different input types available
  3. Go to Settings → Indexes — list the indexes that exist in your environment
  4. Navigate to Settings → Server Settings → General Settings — note the Splunk server name and default index
✅ What you should see

At minimum, the main and _internal indexes. The _internal index stores Splunk's own operational logs — we'll use this heavily for practice.

1.3

Your First Search

Run a simple search and explore the results

  1. In the Search bar, type the following and press Enter:
SPL
index=_internal
  1. Set the Time Range to Last 15 minutes
  2. Observe the Events tab — click on any event to expand it
  3. In the expanded event, identify: host, source, sourcetype, _time
  4. Look at the Fields sidebar on the left — how many fields are listed under "Selected Fields"?
  5. Click the Patterns tab — what common patterns does Splunk identify?
💡 Tip

Every search starts with an index. The _internal index is always available because it contains Splunk's own logs — perfect for practice!

1.4

Understanding Splunk Components

Map the three core processing layers

🎯 Scenario

Your manager asks you to explain how data flows through Splunk. Explore the following to build your understanding.

  1. Run this search to see Splunk's internal data pipeline:
    SPL
    index=_internal sourcetype=splunkd component=Metrics group=pipeline
| head 20
  2. Notice the processor field — this shows data pipeline stages
  3. Now explore how data is indexed:
    SPL
    index=_internal sourcetype=splunkd component=Metrics group=per_index_thruput

| sort - total_KB
  4. This shows data throughput per index — which index is receiving the most data?

Lesson 200.2 — Basic Searching

Master keyword searches, Boolean operators, time ranges, and search modes.

Keywords & PhrasesBoolean OperatorsTime Range PickerSearch Modes
2.1

Keyword & Phrase Searches

Find events using simple text matching

  1. Search for a keyword:
    SPL
    index=_internal error
  2. Search for an exact phrase (use quotes):
    SPL
    index=_internal "failed to connect"
  3. Search for a key-value pair:
    SPL
    index=_internal log_level=ERROR
  4. Compare the result counts from steps 1 and 3 — why might they differ?
✅ Expected

The keyword error matches anywhere in the raw text (case-insensitive). The field log_level=ERROR only matches the specific field. The keyword search typically returns more results because it finds "error" in any context.

2.2

Boolean Operators

Combine search terms with AND, OR, NOT

  1. AND is implied — these are the same:
    SPL
    index=_internal error connection
    SPL
    index=_internal error AND connection
  2. Use OR to broaden:
    SPL
    index=_internal error OR warning
  3. Use NOT to exclude:
    SPL
    index=_internal error NOT debug
  4. Use parentheses for complex logic:
    SPL
    index=_internal (error OR warning) NOT debug
  5. Compare result counts at each step
💡 Tip

Boolean operators must be UPPERCASE: AND, OR, NOT. Lowercase versions are treated as keywords.

2.3

Wildcards

Use * to match partial values

  1. Match anything starting with "fail":
    SPL
    index=_internal fail*
  2. Match anything ending with "Error":
    SPL
    index=_internal sourcetype=splunkd *Error
  3. Use wildcards in field values:
    SPL
    index=_internal component=Index*
💡 Tip

Wildcards at the beginning of a term (*Error) are slower because Splunk can't use the index efficiently. Place wildcards at the end when possible.

2.4

Time Range Picker

Control the time window of your searches

  1. Run a search with Last 60 minutes:
    SPL
    index=_internal sourcetype=splunkd
  2. Note the event count. Now change to Last 15 minutes — how does the count change?
  3. Try Last 4 hours — observe the increase
  4. Use inline time modifiers (these override the picker):
    SPL
    index=_internal sourcetype=splunkd earliest=-30m latest=now
  5. Try a specific date range:
    SPL
    index=_internal sourcetype=splunkd earliest=-7d@d latest=@d
✅ Expected

@d snaps to the beginning of the current day. -7d@d means 7 days ago at midnight. This gives you exactly one week of data.

2.5

Search Modes — Smart, Fast, Verbose

Understand how search modes affect results and performance

  1. Set search mode to Verbose and run:
    SPL
    index=_internal sourcetype=splunkd
  2. Check the Fields sidebar — count the number of fields shown
  3. Switch to Fast mode and re-run the same search — how many fields now?
  4. Switch to Smart mode — run the search again, then run with a transforming command:
    SPL
    index=_internal sourcetype=splunkd | stats count by log_level
  5. Observe: In Smart mode, Splunk automatically adjusts behavior based on whether you use transforming commands
💡 Tip

Fast = fewest fields, fastest. Verbose = all fields, slowest. Smart = best of both — the recommended default.

Lesson 200.3 — Using Fields in Searches

Master the Fields Sidebar, use fields as filters, comparison operators, and the table/rename commands.

Fields SidebarField FiltersComparison Operatorstable & rename
3.1

Explore the Fields Sidebar

Discover Selected Fields, Interesting Fields, and the Field Window

  1. Run:
    SPL
    index=_internal sourcetype=splunkd earliest=-15m
  2. In the Fields Sidebar, identify: Selected Fields (host, source, sourcetype) and Interesting Fields
  3. Click on the component field — the Field Window opens showing top values, count, and percentage
  4. Click "Top values" in the Field Window — observe the SPL auto-generated in the search bar
  5. Try clicking "Top values by time" — what command does Splunk generate?
  6. Click "All Fields" at the bottom of the sidebar — check some new fields to add them to Selected Fields
3.2

Fields as Filters

Use field=value pairs for precise searching

  1. Basic field filter:
    SPL
    index=_internal sourcetype=splunkd log_level=ERROR
  2. Multiple field filters (AND logic):
    SPL
    index=_internal sourcetype=splunkd log_level=ERROR component=DatabaseDirectoryManager
  3. OR within the same field:
    SPL
    index=_internal sourcetype=splunkd (log_level=ERROR OR log_level=WARN)
  4. Exclude with != :
    SPL
    index=_internal sourcetype=splunkd log_level!=INFO
💡 Tip

Field names are case-sensitive (log_level ≠ Log_Level). Field values are NOT (ERROR = error). Always check the Fields Sidebar for exact field names.

3.3

Comparison Operators

Use =, !=, <, >, <=, >= for numeric filtering

  1. Greater than:
    SPL
    index=_internal sourcetype=splunkd linecount > 5
  2. Less than or equal:
    SPL
    index=_internal sourcetype=splunkd linecount <= 1
  3. Combine comparisons:
    SPL
    index=_internal sourcetype=splunkd linecount >= 3 linecount <= 10
  4. Compare result counts at each step
3.4

The table & rename Commands

Create clean, formatted output

  1. Basic table:
    SPL
    index=_internal sourcetype=splunkd log_level=ERROR
| table _time host component message
  2. Table with rename:
    SPL
    index=_internal sourcetype=splunkd log_level=ERROR
| table _time host component message
| rename _time as "Timestamp", host as "Server", component as "Module", message as "Error Details"
  3. Try renaming with spaces in the new name — remember to use quotes!
💡 Tip

After rename, the old field name no longer works in subsequent commands. Always rename LAST in your pipeline.

Lesson 200.4 — SPL Fundamentals

Master the search pipeline, fields command, dedup, sort, and eval to build powerful multi-stage searches.

Search Pipelinefieldsdedupsorteval
4.1

Build a Search Pipeline

Chain commands with the pipe operator

Build this pipeline one step at a time — run after each pipe to see how data transforms:

Step 1: Base search
index=_internal sourcetype=splunkd
Step 2: Add fields
index=_internal sourcetype=splunkd
| fields host component log_level
Step 3: Add dedup
index=_internal sourcetype=splunkd
| fields host component log_level
| dedup host component
Step 4: Add sort
index=_internal sourcetype=splunkd
| fields host component log_level
| dedup host component
| sort host component
Step 5: Add table
index=_internal sourcetype=splunkd
| fields host component log_level
| dedup host component
| sort host component
| table host component log_level
4.2

The fields Command

Include and exclude specific fields

  1. Include only specific fields:
    SPL
    index=_internal sourcetype=splunkd
| fields host source sourcetype log_level
  2. Exclude fields (note the minus sign):
    SPL
    index=_internal sourcetype=splunkd
| fields - _raw punct splunk_server
  3. Compare: Run without fields → check how many fields show in sidebar. Then run with fields — the sidebar changes
💡 Tip

Separating field names? Use spaces, not commas. fields host source ✅ | fields host, source ❌

4.3

The dedup Command

Remove duplicate events

  1. Without dedup — see all events:
    SPL
    index=_internal sourcetype=splunkd | table host component
  2. With dedup — one per combination:
    SPL
    index=_internal sourcetype=splunkd | table host component | dedup host component
  3. Keep 3 per combination:
    SPL
    index=_internal sourcetype=splunkd | table host component | dedup 3 host component
  4. Compare result counts at each step — the difference shows dedup's impact
4.4

The sort Command

Order results ascending and descending

  1. Sort ascending:
    SPL
    index=_internal sourcetype=splunkd
| stats count by component
| sort + count
  2. Sort descending:
    SPL
    index=_internal sourcetype=splunkd
| stats count by component
| sort - count
  3. Sort by multiple fields:
    SPL
    index=_internal sourcetype=splunkd
| stats count by host component
| sort host - count
  4. Limit results:
    SPL
    index=_internal sourcetype=splunkd
| stats count by component
| sort 5 - count
4.5

The eval Command — Calculations

Create new fields with expressions

  1. Create a calculated field:
    SPL
    index=_internal sourcetype=splunkd
| eval event_size_KB = len(_raw) / 1024
| table _time host event_size_KB
  2. Use if() for conditional logic:
    SPL
    index=_internal sourcetype=splunkd
| eval severity = if(log_level=="ERROR", "CRITICAL", "NORMAL")
| table _time host log_level severity
  3. String concatenation with period (.):
    SPL
    index=_internal sourcetype=splunkd
| eval full_source = host . " — " . component
| table _time full_source log_level
  4. The case() function for multiple conditions:
    SPL
    index=_internal sourcetype=splunkd
| eval priority = case(
    log_level=="ERROR", "🔴 High",
    log_level=="WARN", "🟡 Medium",
    log_level=="INFO", "🟢 Low",
    1==1, "⚪ Unknown")
| table _time host priority log_level
4.6

🔥 Challenge: Complete Analysis Pipeline

Chain all 4 commands together

🎯 Scenario

Build a search that shows unique host-component combinations with error counts, sorted by most errors first, with severity labels and clean column names.

Full Pipeline
index=_internal sourcetype=splunkd log_level=ERROR earliest=-24h
| stats count as errors by host component
| eval severity = if(errors > 100, "CRITICAL", "NORMAL")
| sort - errors
| rename host as "Server", component as "Module", errors as "Error Count"

Lesson 200.5 — Transforming Commands

Turn raw events into structured statistical summaries with top, rare, and stats.

toprarestatsStatistical Functions
5.1

The top Command

Find the most common values

  1. Basic top:
    SPL
    index=_internal sourcetype=splunkd | top component
  2. Limit results:
    SPL
    index=_internal sourcetype=splunkd | top limit=5 component
  3. Show all values (limit=0):
    SPL
    index=_internal sourcetype=splunkd | top limit=0 log_level
  4. Top by another field:
    SPL
    index=_internal sourcetype=splunkd | top component by host
  5. Rename the count/percent columns:
    SPL
    index=_internal sourcetype=splunkd | top component countfield="Events" percentfield="Share"
✅ Expected

Results appear on the Statistics tab with count and percent columns. Click Visualization to see charts.

5.2

The rare Command

Find the least common values

  1. Find rarest components:
    SPL
    index=_internal sourcetype=splunkd | rare component
  2. Compare with top:
    SPL
    index=_internal sourcetype=splunkd | rare limit=5 log_level
💡 Tip

In security, rare is gold. Rare user agents, rare source IPs, rare file names — anomalies often indicate threats. If something appears rarely, investigate it.

5.3

The stats Command — Core Functions

count, sum, avg, min, max, dc, values, list

  1. Count events by field:
    SPL
    index=_internal sourcetype=splunkd | stats count by log_level
  2. Count with rename:
    SPL
    index=_internal sourcetype=splunkd | stats count as "Total Events" by host
  3. Multiple functions:
    SPL
    index=_internal sourcetype=splunkd
| stats count as "Events",
        min(linecount) as "Min Lines",
        max(linecount) as "Max Lines",
        avg(linecount) as "Avg Lines"
  by component
| sort - Events
  4. Distinct count (dc):
    SPL
    index=_internal sourcetype=splunkd
| stats dc(component) as "Unique Components" by host
  5. The values() function (multivalue):
    SPL
    index=_internal sourcetype=splunkd
| stats values(log_level) as "Log Levels Seen", count by component
| sort - count

Lesson 200.7 — Creating & Using Lookups

Enrich event data with external context using CSV lookups, definitions, and automatic lookups.

Lookup TypesCSV UploadLookup DefinitionsAutomatic Lookupslookup / inputlookup / outputlookup
7.1

Explore Existing Lookups

Browse lookup files and view their contents

  1. Navigate to Settings → Lookups → Lookup table files
  2. Browse the list — note Path, Owner, App, Sharing
  3. View a lookup file's contents:
    SPL
    | inputlookup geo_attr_countries.csv
  4. Filter lookup contents:
    SPL
    | inputlookup geo_attr_countries.csv | head 20
  5. Search within a lookup:
    SPL
    | inputlookup geo_attr_countries.csv | where lat > 40
7.2

Upload a CSV Lookup File

Complete the upload → permissions → definition workflow

  1. Upload: Settings → Lookups → Lookup table files → + Add new
  2. App: search, select your CSV file, set Destination filename (e.g., http_codes.csv)
  3. Permissions: Find your file → Permissions → This app only → Read for Everyone → Save
  4. Definition: Settings → Lookups → Lookup definitions → + Add new
  5. Name it (e.g., http_codes), select the table file → Save
  6. Verify:
    SPL
    | inputlookup http_codes.csv
💡 Tip

If you don't have a CSV, create a simple one in a text editor with headers: status,status_description,status_type and a few rows like 200,OK,Success.

7.3

Use the lookup Command

Enrich search results with lookup data

  1. Without lookup — raw status codes:
    SPL — Before
    index=web | stats count by status
  2. With lookup — enriched data:
    SPL — After
    index=web
| lookup http_codes.csv status OUTPUT status_description status_type
| stats count by status status_description status_type
  3. Use outputlookup to create a new lookup from search results:
    SPL
    index=_internal sourcetype=splunkd log_level=ERROR
| stats count by component
| outputlookup error_components.csv
  4. Verify what was written:
    SPL
    | inputlookup error_components.csv
7.4

Configure an Automatic Lookup

Enrich data without typing the lookup command

  1. Settings → Lookups → Automatic lookups → + Add new
  2. App: search, Name: http_codes
  3. Lookup table: http_codes, Apply to sourcetype: access_combined
  4. Input fields: status → status
  5. Output fields: status_description, status_type
  6. Save → set permissions to Read for Everyone
  7. Verify — run WITHOUT the lookup command:
    SPL — Auto enrichment
    index=web sourcetype=access_combined
| stats count by status_type, status_description
✅ Expected

The status_type and status_description fields appear automatically — without using the lookup command. That's the power of automatic lookups!

Lesson 200.8 — Scheduled Reports & Alerts

Automate monitoring with scheduled reports, create alerts with trigger conditions, and manage triggered alert investigations.

Scheduled ReportsScheduled AlertsReal-Time AlertsRolling WindowTriggered Alerts
8.1

Create a Scheduled Report

Set up a report that runs automatically

  1. Run a search:
    SPL
    index=_internal sourcetype=splunkd | stats count by log_level
  2. Click Save As → Report
  3. Title: Daily_Log_Level_Report
  4. Click Schedule → Enable schedule
  5. Schedule: Run every day at 08:00
  6. Time range: Last 24 hours
  7. Click Save
  8. Verify: Settings → Searches, reports, and alerts → find your report
💡 Tip

Remember: A scheduled report runs its action EVERY time. A scheduled alert runs its action only when conditions are met.

8.2

Create a Scheduled Alert

Alert when error count exceeds a threshold

  1. Run:
    SPL
    index=_internal sourcetype=splunkd log_level=ERROR
| stats count as error_count
  2. Click Save As → Alert
  3. Title: Training_Alert_Error_Spike
  4. Alert type: Scheduled, run every hour
  5. Trigger condition: Number of Results > 50
  6. Trigger: Once
  7. Action: Add to Triggered Alerts, Severity: High
  8. Click Save
8.3

Create a Real-Time Per-Result Alert

Trigger on every matching event with throttling

  1. Run:
    SPL
    index=_internal sourcetype=splunkd log_level=ERROR
  2. Save As → Alert
  3. Title: RT_Alert_Errors
  4. Alert type: Real-time
  5. Trigger: Per result
  6. Throttle: ✅ Enabled — suppress for 30 minutes
  7. Action: Add to Triggered Alerts, Severity: High
  8. Save → go to Activity → Triggered Alerts to watch it fire
💡 Tip

Without throttling, per-result alerts can generate thousands of notifications. Always set a suppression period in production!

8.4

Create a Rolling Time Window Alert

Detect patterns over a moving time frame

  1. Search:
    SPL
    index=_internal sourcetype=splunkd log_level=ERROR
  2. Save As → Alert
  3. Title: Pattern_Alert_Error_Burst
  4. Alert type: Real-time
  5. Trigger: Number of Results > 20 in 5 minutes
  6. Trigger: Once
  7. Throttle: Suppress for 30 minutes
  8. Action: Add to Triggered Alerts, Severity: Critical
✅ Key Difference

Per-result = every single matching event. Rolling window = only when a PATTERN is detected (20+ events in 5 min). Rolling window detects bursts, not individual events.

8.5

Investigate Triggered Alerts

View, analyze, and manage fired alerts

  1. Go to Activity → Triggered Alerts
  2. Review any alerts that fired — note Time, Type, Severity, Mode
  3. Click View results on a triggered alert to see the event data
  4. Investigate with internal searches:
    SPL — View alert scheduler status
    index=_internal sourcetype=scheduler
| stats count by savedsearch_name status
    SPL — List all fired alerts via REST
    | rest /services/alerts/fired_alerts
| table title, app, alert_type, severity
  5. From the Alerts page, try: Edit, Disable, Clone, and Delete options
